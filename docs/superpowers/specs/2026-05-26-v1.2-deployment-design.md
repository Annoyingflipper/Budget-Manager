# Budget Manager v1.2 — Deployment & QA Environment Spec

**Date:** 2026-05-26
**Status:** Approved — ready for implementation planning
**Builds on:** v1.0 (`docs/superpowers/specs/2026-05-25-budget-manager-design.md`) and v1.1 (`docs/superpowers/specs/2026-05-25-frontend-redesign-design.md`)

## 1. Overview

v1.2 takes the working v1.1 app out of localhost-only and into a proper two-environment deployment on Vercel + Supabase. After v1.2 you'll have:

- A **production** URL (`budget-manager.vercel.app`) backed by your existing Supabase project with real budget data.
- A **QA** URL (`budget-manager-qa.vercel.app`) backed by a new, empty Supabase project — where you push changes first to validate before they hit production.
- Local dev (`npm run dev`) still works for fast iteration and points at the QA Supabase, so day-to-day coding never risks real data.
- A documented workflow (`docs/superpowers/DEPLOYMENT.md`) for shipping changes through QA → PRD safely.

No application source code changes. Everything is infrastructure, env configuration, and process documentation.

## 2. Goals

- Deploy the v1.1 codebase to Vercel as the production environment, backed by the existing Supabase project (`vouazcrsrdnaeivffwot`) that holds real budget data.
- Create a new Supabase project for QA, schema-identical to PRD, seeded empty.
- Set up a single Vercel project where `main` deploys to production and the `staging` branch deploys to a stable preview URL aliased as `budget-manager-qa.vercel.app`.
- Switch local dev to point at the QA Supabase by default so accidental writes during coding don't touch real data.
- Gate Vercel builds behind `npm test` so a failing test suite blocks deploys.
- Document the QA → PRD promotion workflow (code + schema migrations) so future versions follow it consistently.

## 3. Non-goals (v1.2)

- Custom domain — staying on `.vercel.app` subdomains.
- Per-pull-request preview deployments as a formal review workflow. (Vercel will still create them automatically; we just don't depend on them.)
- Automated migration apply (still manual via the Supabase MCP or `supabase db push`).
- A production rollback runbook beyond the one-paragraph notes in this spec.
- GitHub Actions / CI beyond the `npm test` gate inside Vercel's build.
- Cloning real PRD data into QA. QA starts empty.
- Any application source code changes.

## 5. Workflow

Day-to-day from v1.2 onward, every change flows through this path:

1. **Code locally.** `npm run dev` reads `.env.development.local`, which points at QA Supabase. HMR, fast iteration. Real data is never touched.
2. **Schema change (if any).** Add a new migration file in `supabase/migrations/`. Apply it to **QA Supabase** via the Supabase MCP. Verify in the Supabase dashboard.
3. **Push to `staging` branch.** Vercel detects the push, runs `npm test && npm run build`. If green, deploys to the `staging` preview URL aliased as `budget-manager-qa.vercel.app`.
4. **Test on QA.** Open the QA URL in a real browser. Run through whatever the change affected: sign up / log in, edit values, navigate Settings, switch themes, exercise the new feature end-to-end.
5. **Schema change for PRD (if any).** Once QA is green, apply the same migration to **PRD Supabase** via the Supabase MCP.
6. **Merge `staging` → `main`.** Vercel re-runs the build against the merged commit and deploys to production. Tests must pass again.
7. **Smoke test PRD.** Open the production URL. Light verification that nothing regressed in the move from QA to PRD env vars.

When a release is feature-complete, follow the v1.0/v1.1 pattern: add a new sub-page to the Budget Manager hub in Notion with a smoke-test checklist for that version.

## 9. Env var matrix

Five surfaces need env vars; each maps to one of the two Supabase projects:

| Surface | File / dashboard | Points at |
|---|---|---|

## 10. Migration flow

The single most important rule: **migrations always go to QA first, then PRD.** A bad migration applied straight to PRD could corrupt your real budget data; QA is the safety net.

```
write migration file ──→ apply to QA via MCP ──→ test on QA URL ──→ apply to PRD via MCP ──→ merge to main
        │                       │                      │                      │                    │
        local                   QA Supabase            QA Vercel              PRD Supabase         PRD Vercel
```

Concretely:

1. Author the SQL in `supabase/migrations/000N_<name>.sql`. Commit it on the `staging` branch.
2. Apply to QA via `mcp__plugin_supabase_supabase__apply_migration` with `project_id` = QA ref.
3. Verify QA via `list_tables`, `list_migrations`, `get_advisors`. All clean.
4. Push the code/migration to `staging`. Vercel deploys to QA URL.
5. Smoke-test the change on QA URL.
6. Once green, apply the same migration to PRD via `apply_migration` with `project_id` = `vouazcrsrdnaeivffwot`.
7. Merge `staging` → `main`. Vercel deploys to production.

The Supabase MCP records each `apply_migration` call in the project's migration history table; running it twice with the same name is rejected, so accidental double-apply is prevented.

## 11. Branching

- **`main`** — protected by convention (never push directly except for typo fixes). Source of truth for what's deployed to production.
- **`staging`** — created from `main` at v1.2 setup. All day-to-day development merges land here. Vercel deploys this branch to the QA URL.
- **Feature branches** — optional for solo dev. If used, merge into `staging` (not `main`). Each feature branch automatically gets a Vercel preview URL too, but we don't depend on those for QA.

After v1.2 setup, the first action in any new development task is `git checkout staging && git pull`.

## 12. Testing strategy

- The existing 32 unit/component tests run on every Vercel build via `npm test` in `buildCommand`. A red test halts the deploy.
- The smoke-test pattern from v1.0/v1.1 continues: a new sub-page under Notion → Budget Manager → "v1.2 Smoke Tests" enumerates the checklist for this release.
- No new automated tests in v1.2. The deliverable is infrastructure, not behavior, so the existing tests + manual smoke tests on QA are the gate.

## 13. Failure modes and recovery

**Vercel build fails on `staging` push.** No deploy happens. Inspect the build log in the Vercel dashboard. Common causes: test failure, env var typo, package-lock drift. Fix locally, push again.

**QA app loads but Supabase calls fail.** Almost always an env-var mismatch in Vercel's Preview environment. Verify the values in **Project Settings → Environment Variables → Preview** match the QA Supabase project URL + anon key exactly.

**Migration applied to QA broke something.** Don't apply to PRD. Write a follow-up migration to revert or fix on QA, then re-apply forward.

**Migration applied to PRD broke something.** Two options: (a) write a follow-up forward-fix migration on QA, validate, apply to PRD; (b) Supabase point-in-time restore on the PRD project (dashboard → Database → PITR), which rolls back the whole DB to a chosen timestamp. PITR is destructive of recent legitimate changes, so option (a) is preferred unless data is genuinely corrupted.

**Production deploy broke something.** In Vercel dashboard → Deployments → find the last-known-good deployment → "Promote to Production". Instant rollback. Then debug the broken commit on `staging`.

**QA and PRD schemas drift** (e.g., a migration applied to one but not the other). Use `mcp__plugin_supabase_supabase__list_migrations` on both project IDs and diff the lists. Re-apply the missing migration to the lagging side.

## Appendix A — Vercel preview URL naming

When Vercel deploys a branch, it generates a URL of the form:
```
<project-name>-git-<branch-name>-<username-or-team>.vercel.app
```

For this project pushing the `staging` branch, the auto-generated URL would be:
```
budget-manager-git-staging-<username>.vercel.app
```

We then add `budget-manager-qa.vercel.app` as a domain alias in Vercel settings, configured to track deployments of the `staging` branch. The alias is free; the underlying URL still exists but we don't share it.

If the project name `budget-manager` is taken on Vercel, suffix it (e.g., `budget-manager-flipper`) — both URLs adjust accordingly.

## Appendix B — Why anon keys are safe to put in env vars and ship to the browser

Supabase's anon (publishable) key is a JWT that grants access to the API but is gated by Row-Level Security at the database level. Every table in this project has RLS enabled with policies requiring `auth.uid() = user_id AND aal = 'aal2'`. Without a signed-in MFA-completed session, the anon key can read or write nothing. Putting it in the bundled JS is exactly how Supabase is designed to be deployed.

The *service-role* key is different — it bypasses RLS. We never use it client-side, never put it in Vercel, never commit it. v1.2 doesn't introduce any service-role usage.
