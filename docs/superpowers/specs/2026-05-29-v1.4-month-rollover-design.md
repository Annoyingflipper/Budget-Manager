# Budget Manager v1.4 — Month Rollover & History View Spec

**Date:** 2026-05-29
**Status:** Approved — ready for implementation planning

---

## 1. Overview

v1.4 adds a **month dimension** to the budget. Until now the app has stored one income row and one set of line items per user — a single, perpetual "current budget." v1.4 attaches a `period_month` to that data so each calendar month is its own snapshot, lets you navigate between months with back/forward arrows in the header, and adds an explicit **"Start \<next month\>"** button that copies line item names and projected values forward (resetting actuals to zero) when you're ready to begin a new month.

Past months remain fully editable so you can catch up on receipts or fix typos. No data is ever deleted automatically. Categories stay user-level (the eight seeded categories don't vary by month) — per-month custom categories is v1.5.

## 2. Goals

- Add a `period_month date NOT NULL` column to `income` and `line_items`. Always the 1st of the month, enforced by a `CHECK` constraint.
- Backfill all existing rows with `period_month = 2026-06-01` (the user's current data is their June 2026 plan).
- Provide a `rolloverMonth(fromMonth, toMonth)` Postgres RPC that, in one transaction, copies the prior month's line item names + projected values into the new month with actuals zeroed.
- Header switcher: `← June 2026 →`. The right arrow becomes a **"Start July 2026"** button when you're viewing the most recent month with data.
- Past months are fully editable through the existing CRUD path; nothing in the UI gates edits by month.
- Empty future months show standard empty-category cards and a zeroed income summary; first save lazily inserts rows for that month.

## 3. Non-goals (v1.4)

- **Per-month custom categories.** Categories remain user-level. Custom user-editable categories (rename / add / delete / reorder) is v1.5.
- **Locking a past month.** All months stay editable. No "close month" workflow.
- **Month dropdown picker.** Back/forward arrows only in v1.4. Revisit if/when the user accumulates enough months that scrolling hurts.
- **Inter-month analytics or charts.** A month is a self-contained view. Comparisons across months are v1.5+.
- **Undo for rollover.** A confirmation dialog precedes the action; reverting means deleting the new month's rows by hand in Supabase. Acceptable for a personal app.
- **"Copy from a specific past month"** — rollover always copies from the month you're viewing. No month-picker on the rollover dialog.
- **Migration of the seeded income row format.** New users currently get an `income` row seeded on signup; v1.4 drops that insertion entirely (it would otherwise need a `period_month`). First save lazily inserts the row for the user's first month.

## 5. Schema (the SQL)

### 5.1 `0009_period_month.sql`

```sql
-- income: add period_month, backfill, repoint primary key
alter table public.income add column period_month date;
update public.income set period_month = date '2026-06-01' where period_month is null;
alter table public.income alter column period_month set not null;
alter table public.income drop constraint income_pkey;
alter table public.income add constraint income_pkey primary key (user_id, period_month);
alter table public.income
  add constraint income_period_first_of_month
  check (period_month = date_trunc('month', period_month)::date);

-- line_items: add period_month, backfill, index
alter table public.line_items add column period_month date;
update public.line_items set period_month = date '2026-06-01' where period_month is null;
alter table public.line_items alter column period_month set not null;
create index line_items_user_period_idx on public.line_items (user_id, period_month);
alter table public.line_items
  add constraint line_items_period_first_of_month
  check (period_month = date_trunc('month', period_month)::date);
```

### 5.2 `0010_seed_trigger_no_income.sql`

Replace the `handle_new_user` trigger body so it inserts only categories — no income row. New users land on the current real-world month with zeros until they save.

### 5.3 `0011_rollover_rpc.sql`

```sql
create or replace function public.rollover_month(from_month date, to_month date)
returns void
language plpgsql
security invoker
set search_path = public
as $$
declare
  uid uuid := auth.uid();
begin
  if uid is null then
    raise exception 'not authenticated';
  end if;
  if (select auth.jwt() ->> 'aal') <> 'aal2' then
    raise exception 'mfa required';
  end if;
  if from_month is null or to_month is null then
    raise exception 'from_month and to_month required';
  end if;
  if from_month <> date_trunc('month', from_month)::date
     or to_month <> date_trunc('month', to_month)::date then
    raise exception 'months must be the first of the month';
  end if;
  if exists (select 1 from public.income where user_id = uid and period_month = to_month)
     or exists (select 1 from public.line_items where user_id = uid and period_month = to_month) then
    raise exception 'month % already has data; cannot roll over into it', to_month;
  end if;

  insert into public.income (user_id, period_month, projected, actual)
  select uid, to_month, projected, 0
  from public.income
  where user_id = uid and period_month = from_month;

  insert into public.line_items (user_id, category_id, name, projected, actual, period_month)
  select uid, category_id, name, projected, 0, to_month
  from public.line_items
  where user_id = uid and period_month = from_month;
end;
$$;

grant execute on function public.rollover_month(date, date) to authenticated;
```

Idempotent in the sense that a second call for the same `to_month` raises rather than duplicating rows. Wrapped in an implicit transaction by Postgres for the function body.

## 6. API surface (`src/api/budget.ts`)

```ts
// Returns budget for a specific month. If no income row exists for (user, month),
// returns income = { projected: 0, actual: 0 } as a virtual default.
async function getBudget(periodMonth: string): Promise<Budget>

// Returns distinct period_month values for this user, sorted descending.
// Sources: union of income.period_month and line_items.period_month.
// Used by App on initial load to pick the default month.
async function listMonths(): Promise<string[]>

// Wraps a single .rpc('rollover_month', { from_month, to_month }) call.
async function rolloverMonth(fromMonth: string, toMonth: string): Promise<void>

// Existing helpers gain periodMonth and stamp it on writes:
async function updateIncome(periodMonth: string, patch: Partial<Income>): Promise<void>
async function addLineItem(periodMonth: string, categoryId: number, name: string, projected: number, actual: number): Promise<LineItem>
async function updateLineItem(periodMonth: string, id: number, patch: Partial<LineItem>): Promise<void>
async function removeLineItem(id: number): Promise<void>  // id is unique; no month needed
```

`updateIncome` is upsert-shaped: if no row exists for `(user, periodMonth)`, it inserts; otherwise updates. The Supabase JS client's `upsert` with `onConflict: 'user_id,period_month'` covers it.

## 7. UI

### 7.1 Header switcher

```
[ ← ]  June 2026  [ → ]            ⚙ Settings   Log out
```

- `←` always enabled. Clicking decrements the month by one calendar month. If no data exists for that month, the view renders empty categories + zero income (same as a brand-new month). Edits there work and lazily create rows on save.
- `→` becomes a **"Start \<next month\>"** button when `selectedMonth >= latestMonth`. Otherwise it's a plain arrow that moves to `selectedMonth + 1 month`.
- The label is plain text in v1.4. No dropdown picker.
- Mobile (≤640px): arrows and label stack above the BalanceHero card on their own row to keep them tappable.

### 7.2 Rollover dialog

Triggered by clicking "Start \<next month\>". A confirmation dialog:

> **Start July 2026?**
> This copies item names and projected values from June 2026. Actuals start at zero.
> [Cancel] [Start July 2026]

On confirm: call `rolloverMonth(selectedMonth, nextMonth)`; on success, set `selectedMonth = nextMonth` and re-fetch budget. On RPC error, show the error message and leave the user on `selectedMonth`.

### 7.3 Empty future month (manual rollover path)

If you `→` past your latest month without rolling over, you see the existing `EmptyCategoryCard`s under each category and a zeroed `IncomeSummary`. Typing into them follows the existing add-item / save-on-blur paths, which lazily create rows tagged with `selectedMonth`.

### 7.4 Initial load

`App.tsx` on mount:

```ts
const months = await listMonths();
const initial = months[0] ?? formatMonth(new Date());  // YYYY-MM-01
setSelectedMonth(initial);
const budget = await getBudget(initial);
setBudget(budget);
```

`formatMonth(d)` returns `${YYYY}-${MM}-01`.
