# Budget Manager v1.3 — Flexible MFA Spec

**Date:** 2026-05-26
**Status:** Approved — ready for implementation planning
**Builds on:** v1.0 (MFA via TOTP), v1.1 (Settings page, ThemeProvider), v1.2 (PRD/QA Vercel + Supabase).

---

## 1. Overview

v1.3 turns MFA from "one mandatory TOTP factor" into "pick the factor(s) you want." Users can enroll a WebAuthn passkey (Touch ID / Face ID / Windows Hello / hardware key), a TOTP authenticator app, or both. At sign-in, they're challenged with whichever factor they have — and if they have both, they pick which to use this time.

Two smaller items ship in the same release as folded-in backlog: the leaked-password protection toggle (a Supabase Auth dashboard setting) and the missing component tests (CategoryTable, IncomeSummary, GrandTotals) flagged in earlier code reviews.

The schema doesn't change. Existing TOTP-only users keep working exactly as today; they can add a passkey later from a new Settings → Security section.

## 2. Goals

- Add WebAuthn passkey as a supported MFA factor type alongside TOTP.
- Let a user have 1 or 2 enrolled factors (one of each type). At least one factor is required to access the app.
- Adapt the sign-in MFA challenge to the user's enrolled factors: if both exist, the user picks per session.
- Add a Settings → Security section to list, add, and remove factors with lockout prevention.
- Enable Supabase's "Leaked Password Protection" on both QA and PRD Auth projects.
- Close the three component test gaps surfaced in the v1.0/v1.1 reviews: CategoryTable add/discard, IncomeSummary save-on-blur, GrandTotals math.
- Maintain the v1.2 deployment workflow (QA first → PRD) for every change.

## 3. Non-goals (v1.3)

- "Trust this device for 30 days" cookie pattern. Passkey + biometric *is* the trusted-device pattern; we don't build a parallel cookie scheme.
- Extending Supabase JWT session lifetime beyond defaults.
- Managing multiple passkeys (or multiple TOTP factors) per account. One of each type is the documented happy path. The UI prevents enrolling a second factor of the same type for v1.3.
- Recovery codes / lost-factor recovery flow. Manual recovery via Supabase dashboard if needed; codes can come in v1.5+ if real users surface the need.
- SMS / phone MFA. Out of scope.
- Any UI for multiple devices, device names beyond the friendlyName Supabase assigns.
- Charts, month rollover, custom categories, sharing — all unchanged from earlier deferred lists.

## 4. Tech stack additions

| Concern | Choice |
|---|---|
| WebAuthn ceremony | Browser-native `navigator.credentials.create()` / `.get()`. No third-party library. |
| Supabase SDK | Existing `@supabase/supabase-js` v2 — its `mfa.enroll({ factorType: 'webauthn' })` and `mfa.challenge` / `mfa.verify` flows handle the server side. |
| Tests | Existing Vitest + RTL. Mock `navigator.credentials` for unit tests. |

No new dependencies. No schema migrations.

## 5. Pre-flight on Supabase (controller-driven, one-time per project)

Both QA and PRD need:

1. **Authentication → Multi-Factor Authentication → enable WebAuthn factor type.** TOTP is already on. We need WebAuthn for the new factor type to be acceptable by `mfa.enroll`.
2. **Authentication → Password Security → enable "Leaked Password Protection".** Closes the existing advisor warning. Supabase will reject sign-ups whose passwords appear in HaveIBeenPwned.
3. Run `get_advisors` afterwards on both projects. The `auth_leaked_password_protection` warning should disappear.

These are not SQL migrations; they're auth config changes. No code in `supabase/migrations/`.

## 6. User flows

### First-time signup (new user)

```
1. Email + password → Supabase sends verification email
2. Click verify link → return to app → log in (email + password)
3. AuthGate state: needs-enroll (currentLevel=aal1, nextLevel=aal1)
4. NEW: FactorPicker screen
   ┌─────────────────────────────────────┐
   │ Add a second factor                 │
   │                                     │
   │ At least one factor required.       │
   │                                     │
   │ [🔐 Passkey                       ] │
   │ [📱 Authenticator app             ] │
   └─────────────────────────────────────┘
5a. User picks Passkey → PasskeyEnroll flow
    - calls mfa.enroll({factorType:'webauthn', friendlyName})
    - browser prompts Touch ID / Face ID / hardware key
    - on success, mfa.challenge + mfa.verify completes enrollment
    - AAL session bumps to aal2
5b. User picks Authenticator app → MFAEnroll TOTP flow (existing)
6. Optional: "Add another factor?" prompt (one shown, the other selectable)
7. AAL2 → AuthGate routes to budget
```

### Returning user (any combination of enrolled factors)

```
1. Email + password
2. AuthGate state: needs-challenge (currentLevel=aal1, nextLevel=aal2)
3. MFAChallenge screen reads listFactors() and branches:
   - Only TOTP enrolled → current TOTP code input
   - Only passkey enrolled → "Tap to unlock with passkey" → triggers navigator.credentials.get()
   - Both enrolled → FactorPicker (challenge mode):
       ┌─────────────────────────────────┐
       │ How do you want to sign in?     │
       │                                 │
       │ [🔐 Use passkey  (recommended)] │
       │ [📱 Use authenticator code   ]  │
       └─────────────────────────────────┘
4. Challenge succeeds → AAL2 → budget
```

### Settings → Security (new section)

Below the existing "Appearance" section:

```
Security
─────────────────────────────────────────────────────────
Two-factor authentication
  🔐 Passkey · "MacBook Pro · 2026-06-01"           [Remove]
  📱 Authenticator app · "iPhone Authenticator"     [Remove]
  [+ Add another factor]
─────────────────────────────────────────────────────────
```

- The "+ Add another factor" button is disabled if the user already has BOTH types enrolled (since v1.3 doesn't allow multiple of one type).
- "Remove" prompts a confirmation. If removing this factor would leave the user with zero verified factors, the action is blocked client-side with a clear error message.
- The page itself only renders if `aal === 'aal2'`. AuthGate already enforces AAL2 to reach Settings.

## 8. New components — interfaces

### `src/api/factors.ts`

```ts
import { supabase } from '../lib/supabase';

export type FactorType = 'totp' | 'webauthn';

export type EnrolledFactor = {
  id: string;
  type: FactorType;
  friendlyName: string;
  createdAt: string;
};

export async function listFactors(): Promise<EnrolledFactor[]>;

export async function enrollTotp(friendlyName: string): Promise<{
  factorId: string;
  qrCode: string;   // SVG data URI
  secret: string;   // for manual entry
}>;

export async function verifyTotpEnrollment(args: {
  factorId: string;
  code: string;
}): Promise<void>;

export async function enrollPasskey(friendlyName: string): Promise<{
  factorId: string;
  credentialCreationOptions: PublicKeyCredentialCreationOptions;
}>;

export async function verifyPasskeyEnrollment(args: {
  factorId: string;
  credential: PublicKeyCredential;
}): Promise<void>;

export async function challengeTotp(factorId: string): Promise<{ challengeId: string }>;
export async function verifyTotpChallenge(args: { factorId: string; challengeId: string; code: string }): Promise<void>;

export async function challengePasskey(factorId: string): Promise<{
  challengeId: string;
  credentialRequestOptions: PublicKeyCredentialRequestOptions;
}>;
export async function verifyPasskeyChallenge(args: {
  factorId: string;
  challengeId: string;
  credential: PublicKeyCredential;
}): Promise<void>;

export async function unenrollFactor(factorId: string): Promise<void>;
```

All wrappers thin around `supabase.auth.mfa.*`. Each throws on Supabase error.

### `src/auth/FactorPicker.tsx`

A presentational component with two modes:
- `mode="enroll"` — "Add a second factor" with two big buttons (Passkey / TOTP). Each button is **disabled** if that factor type is already enrolled.
- `mode="challenge"` — "How do you want to sign in?" with the enrolled factor types as options. Passkey is the primary CTA if both are available.

Props:
```ts
type Props = {
  mode: 'enroll' | 'challenge';
  enrolledTypes: FactorType[];   // empty for fresh enroll, 1-2 for challenge
  onPick: (type: FactorType) => void;
};
```

### `src/auth/PasskeyEnroll.tsx`

Owns the WebAuthn ceremony:
1. Call `enrollPasskey()` to get `credentialCreationOptions`.
2. Call `navigator.credentials.create({ publicKey: credentialCreationOptions })`.
3. On success: call `verifyPasskeyEnrollment()`.
4. On user cancel or error: surface a friendly message and offer Retry.

Props:
```ts
type Props = {
  onDone: () => void;       // factor verified; AuthGate observes AAL2
  onCancel: () => void;     // back to picker / back to settings
};
```

Browser-support guard: if `!window.PublicKeyCredential || !navigator.credentials`, render "This browser doesn't support passkeys" with a fallback link to TOTP enrollment.

### `src/auth/PasskeyChallenge.tsx`

Mirror of `PasskeyEnroll`:
1. Call `challengePasskey(factorId)` to get options.
2. Call `navigator.credentials.get({ publicKey: credentialRequestOptions })`.
3. Call `verifyPasskeyChallenge()`.
4. Handle cancel / error with retry.

### `src/components/FactorList.tsx`

Renders the rows in Settings → Security. Reads from `listFactors()`. Provides:
- One row per enrolled factor (icon + name + date + Remove button).
- A "+ Add another factor" button. Disabled when both types are enrolled.
- The Remove button shows a confirmation: "Remove this factor? You'll need [the other factor] to sign in." Greyed out if it would leave 0 factors.

## 9. MFA Enrollment redesign (`src/auth/MFAEnroll.tsx`)

Replaces the current always-TOTP enrollment.

Pseudocode:
```tsx
function MFAEnroll() {
  const enrolled = useFactors();   // hook: subscribes to listFactors result
  const enrolledTypes = enrolled.map(f => f.type);
  const [picked, setPicked] = useState<FactorType | null>(null);

  if (enrolled.length >= 1 && /* AAL2 already from another factor? no, since this is needs-enroll */) {
    // shouldn't reach here
  }

  if (!picked) {
    return <FactorPicker mode="enroll" enrolledTypes={enrolledTypes} onPick={setPicked} />;
  }

  if (picked === 'webauthn') {
    return <PasskeyEnroll onDone={() => setPicked(null)} onCancel={() => setPicked(null)} />;
  }

  return <TotpEnroll onDone={() => setPicked(null)} onCancel={() => setPicked(null)} />;
}
```

The TOTP enrollment body becomes the new `TotpEnroll` component (extracted verbatim from the current `MFAEnroll` body, only wrapped in the new `onDone` / `onCancel` props contract).

After any successful enrollment, the user reaches AAL2 → AuthGate routes to budget. The flow does NOT force enrolling both factors; one is enough.

## 10. MFA Challenge redesign (`src/auth/MFAChallenge.tsx`)

Replaces the current always-TOTP challenge.

Pseudocode:
```tsx
function MFAChallenge() {
  const enrolled = useFactors();
  const enrolledTypes = useMemo(() => new Set(enrolled.map(f => f.type)), [enrolled]);
  const [picked, setPicked] = useState<FactorType | null>(null);

  useEffect(() => {
    // If only one type enrolled, skip the picker
    if (enrolledTypes.size === 1) setPicked([...enrolledTypes][0]);
  }, [enrolledTypes]);

  if (!picked) {
    return <FactorPicker mode="challenge" enrolledTypes={[...enrolledTypes]} onPick={setPicked} />;
  }

  const factor = enrolled.find(f => f.type === picked)!;
  return picked === 'webauthn'
    ? <PasskeyChallenge factorId={factor.id} onSwitch={() => setPicked(null)} />
    : <TotpChallenge factorId={factor.id} onSwitch={() => setPicked(null)} />;
}
```

The TOTP challenge body becomes the new `TotpChallenge` component (extracted from current `MFAChallenge`). `onSwitch` lets the user back out to the picker if they have multiple factors.

## 13. Lockout prevention

The single rule: **a user must always have at least one verified factor** (otherwise RLS denies all access and they can't sign in productively).

Enforcement:

- `unenrollFactor` wrapper checks `listFactors()` first. If the factor being removed is the only verified factor, throws a `LastFactorError` before calling Supabase.
- `FactorList` UI: the Remove button is disabled (with tooltip "You need at least one factor enrolled") when there's only one factor left.
- Server-side, Supabase's MFA admin endpoints would technically allow removing the last factor, but only from a service-role context — the client SDK's `mfa.unenroll` requires AAL2 and would still leave a working session. Even so, our client guard is the primary fence.
- If a user somehow does lock themselves out (e.g., loses both devices), the recovery path is manual: contact the project owner, who can clear the user's factors via the Supabase dashboard. We won't build a self-serve recovery flow in v1.3.

## Appendix A — Why no schema migration

WebAuthn factors live in Supabase's `auth.mfa_factors` table, which already supports `factor_type: 'webauthn'`. Enabling it is an Auth project setting, not a DDL change. Our `public.user_preferences` table, `public.income`, `public.categories`, `public.line_items` are all untouched in v1.3.

## Appendix B — Browser support for WebAuthn

Modern browsers (Chrome 67+, Firefox 60+, Safari 13+, Edge 18+) support `navigator.credentials.create()` / `.get()` with platform authenticator. iOS 16+ and macOS Ventura+ support passkeys via Touch ID / Face ID. Windows 11 supports Windows Hello. Hardware security keys (YubiKey, etc.) work cross-platform.

The `PasskeyEnroll` and `PasskeyChallenge` components detect support via `window.PublicKeyCredential` and `navigator.credentials`. If absent, they render a fallback "Your browser doesn't support passkeys. Use the Authenticator app option instead."

## Appendix C — Failure modes worth knowing

- **User cancels the Touch ID prompt.** `navigator.credentials.create/get` throws a `NotAllowedError`. We catch and show "Cancelled — try again."
- **User has neither factor available.** Lost both phone and laptop. Recovery is manual via Supabase dashboard (owner clears factors, user re-enrolls).
- **Supabase project doesn't have WebAuthn enabled.** `mfa.enroll({factorType:'webauthn'})` returns a 400 / "factor type not enabled" error. We surface this clearly in dev so the project owner knows to flip the toggle.
- **A new passkey doesn't recognize the older session.** Not an issue in v1.3 since enrollment immediately produces an AAL2 session via the new factor.
- **User locks themselves out by removing the wrong factor.** Prevented by the `LastFactorError` check, but if it ever happens, dashboard recovery.
