# Budget Manager v1.3 — Auth Polish & Test Gaps Spec

**Date:** 2026-05-26
**Status:** Approved — ready for implementation planning
**Supersedes:** an earlier v1.3-flexible-mfa-design.md that was abandoned once we discovered Supabase Auth doesn't currently support WebAuthn as an MFA factor type (only `totp` and `phone` are documented in `mfa.enroll`).

---

## 1. Overview

v1.3 is intentionally a small release: rename the TOTP factor's issuer so authenticator-app entries are labeled by environment (QA vs PRD) instead of the meaningless `localhost:3000`, fill three component test gaps surfaced in earlier reviews, and document the current session-persistence behavior so future-you remembers why "Supabase already remembers me."

No new features. No schema changes. No new components. The Settings page is unchanged. The auth flow shape is unchanged — only the label generated at TOTP enrollment.

This release is honest about the platform's current capabilities — WebAuthn/passkey work is deferred until Supabase ships it natively, and leaked-password protection is deferred until we're on Supabase's Pro plan. The user's original request ("remember me on return visits") is already satisfied by Supabase's default session persistence; we document that rather than add UI that wouldn't change behavior.

## 2. Goals

- Rename the TOTP factor's issuer at enrollment so Google Authenticator (and other authenticator apps) show **`budget-manager-qa.vercel.app: <email>`** for QA enrollments and **`budget-manager-drab.vercel.app: <email>`** for PRD enrollments — replacing the current `localhost:3000: <email>` labels that originate from the Supabase project's default Site URL.
- Close the three component test gaps flagged in earlier reviews:
  - `CategoryTable` — empty state, add-item flow, Escape-to-discard
  - `IncomeSummary` — save-on-blur and revert-on-error
  - `GrandTotals` — math, empty-state, color semantics
- Document the current session-persistence model and the deferred items (passkey, leaked-password protection) in `DEPLOYMENT.md` so future contributors don't have to rediscover what's supported.

## 3. Non-goals (v1.3)

- **Leaked-password protection.** Supabase's HaveIBeenPwned toggle is gated to Pro-plan projects; we're on the free tier. Documented as a deferred item; revisit if/when we upgrade.
- **WebAuthn / passkey as an MFA factor.** Supabase Auth doesn't currently support `factorType: 'webauthn'` in `mfa.enroll`. Tracked as a deferred item; revisit when Supabase ships it.
- **"Remember me" checkbox in the UI.** Supabase already persists sessions in `localStorage` with auto-refresh by default; adding a checkbox would be cosmetic. Document the existing behavior in `DEPLOYMENT.md` instead.
- **Custom trusted-device cookie / RLS exception scheme.** Real engineering, not worth the security tradeoff for a personal budgeting app.
- **Phone (SMS) as a second factor.** Supabase supports it, but it costs money per message and is no more biometric than TOTP. No clear win over TOTP.
- **Settings → Security section.** Not built in v1.3 — there's only one factor type to manage today (TOTP), and the existing MFA enrollment flow during sign-up handles it.
- **Rewriting existing TOTP factors.** The issuer is baked into the QR code at enroll time; existing `localhost:3000` entries in the authenticator stay there until the user removes the factor and re-enrolls. Out of scope to migrate them automatically.
- **`src/auth/FactorPicker.tsx`, `PasskeyEnroll.tsx`, `PasskeyChallenge.tsx`, `useFactors.ts`, factor management UI** — none of this lands in v1.3.
- **Migration files.** No SQL changes.

## 7. Documentation changes (`DEPLOYMENT.md`)

Append a new section after "Rollback":

```markdown
## Session lifetime & "remember me" behavior

Supabase persists each user's session to `localStorage` with auto-refresh, so users stay signed in across browser restarts by default — no "remember me" checkbox needed. The relevant configuration:

| Setting | Value | Where |
|---|---|---|
| Access token (JWT) expiry | default (~1h) | Supabase Auth → JWT Settings |
| Refresh token lifetime | default (very long, multi-month) | Supabase Auth → Sessions |
| AAL2 (MFA verification) | preserved across refresh until explicit sign-out | Implicit in Supabase JWT claims |

A user is re-MFA-challenged only when they sign out explicitly, when the refresh token expires (rare), or when their session is cleared (incognito mode, manual localStorage wipe, etc.). If you want to sign out, click "Log out" in the header — otherwise you'll stay at AAL2 for weeks.

## TOTP issuer (authenticator-app label)

Each environment registers its TOTP factor with a different `issuer`, which is the leading label in Google Authenticator / 1Password / etc. The issuer comes from `VITE_APP_URL`, set per Vercel env scope:

| Env | `VITE_APP_URL` | Label in authenticator |
|---|---|---|
| QA (Vercel Preview, `staging` branch) | `https://budget-manager-qa.vercel.app` | `budget-manager-qa.vercel.app: <email>` |
| PRD (Vercel Production, `main` branch) | `https://budget-manager-drab.vercel.app` | `budget-manager-drab.vercel.app: <email>` |
| Local dev | matches `.env.development.local` | same as QA (default) |

The issuer is baked into the QR code at enroll time; changing `VITE_APP_URL` only affects future enrollments. To rebrand an existing factor, unenroll it in Supabase Auth → Users → (user) → Factors and re-enroll on next sign-in.

## Passkey / WebAuthn — deferred

Supabase Auth currently supports only `totp` and `phone` as MFA factor types. Native WebAuthn (passkey) support is being worked on but not yet GA. We intentionally don't build custom WebAuthn or a trusted-device cookie scheme; we'll revisit when Supabase ships native support.

## Leaked-password protection — deferred (Pro plan)

Supabase's HaveIBeenPwned integration (rejects sign-ups using known-breached passwords) is gated to Pro-plan projects. Both QA and PRD are on the free tier, so the toggle in Auth → Providers → Email is read-only for us. Revisit if/when we upgrade.
```

## Appendix A — Why the abandoned plan didn't work

The earlier `v1.3-flexible-mfa-design.md` assumed Supabase's `mfa.enroll` accepted `factorType: 'webauthn'`. A quick `search_docs` lookup of the current docs confirmed it doesn't — the documented factor types are `totp` and `phone` only. The plan would have failed at the first API call.

The honest version of this work waits for native Supabase support and ships the smaller, real items now.

## Appendix B — What the user originally asked

> *"i want to add a remember me check in the login part of the app so whenever i want to come back i don't have to log in from the start."*

After clarification, this turned out to mean "skip the MFA code on return visits for a while." Supabase's default session behavior already satisfies this: AAL2 is preserved across token refreshes; the user only re-MFAs after an explicit sign-out or refresh-token expiry. Documenting this in `DEPLOYMENT.md` (Section 7 above) closes the loop without building UI that wouldn't change behavior.
