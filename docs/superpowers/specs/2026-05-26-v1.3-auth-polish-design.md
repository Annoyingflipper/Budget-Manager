## 1. Overview

v1.3 is intentionally a small release: enable Supabase's leaked-password protection, bump access-token lifetime so sign-ins feel persistent for longer, fill three component test gaps surfaced in earlier reviews, and document the current session-persistence behavior so future-you remembers why "Supabase already remembers me."

No new features. No schema changes. No new components. The Settings page is unchanged. The MFA flow is unchanged.

This release is honest about the platform's current capabilities â€” WebAuthn/passkey work is deferred until Supabase ships it natively. The user's original request ("remember me on return visits") is already satisfied by Supabase's default session persistence; we document that rather than add UI that wouldn't change behavior.

## 2. Goals

- Enable Supabase Auth's leaked-password protection on both QA and PRD projects so new sign-ups against breached passwords are rejected.
- Extend the access-token JWT expiry from the default 1 hour to 24 hours on both projects, reducing the frequency of silent refresh calls.
- Close the three component test gaps flagged in earlier reviews:
  - `CategoryTable` â€” empty state, add-item flow, Escape-to-discard
  - `IncomeSummary` â€” save-on-blur and revert-on-error
  - `GrandTotals` â€” math, empty-state, color semantics
- Document the current session-persistence model and the deferred items in `DEPLOYMENT.md` so future contributors don't have to rediscover what's supported.

## 3. Non-goals (v1.3)

- **WebAuthn / passkey as an MFA factor.** Supabase Auth doesn't currently support `factorType: 'webauthn'` in `mfa.enroll`. Tracked as a deferred item; revisit when Supabase ships it.
- **"Remember me" checkbox in the UI.** Supabase already persists sessions in `localStorage` with auto-refresh by default; adding a checkbox would be cosmetic. Document the existing behavior in `DEPLOYMENT.md` instead.
- **Custom trusted-device cookie / RLS exception scheme.** Real engineering, not worth the security tradeoff for a personal budgeting app.
- **Phone (SMS) as a second factor.** Supabase supports it, but it costs money per message and is no more biometric than TOTP. No clear win over TOTP.
- **Settings â†’ Security section.** Not built in v1.3 â€” there's only one factor type to manage today (TOTP), and the existing MFA enrollment flow during sign-up handles it.
- **`src/auth/FactorPicker.tsx`, `PasskeyEnroll.tsx`, `PasskeyChallenge.tsx`, `useFactors.ts`, factor management UI** â€” none of this lands in v1.3.
- **Migration files.** No SQL changes.

## Session lifetime & "remember me" behavior

Supabase persists each user's session to `localStorage` with auto-refresh, so users stay signed in across browser restarts by default â€” no "remember me" checkbox needed. The relevant configuration:

| Setting | Value | Where |
|---|---|---|
| Access token (JWT) expiry | 86400s (24h) | Supabase Auth â†’ JWT Settings |

| AAL2 (MFA verification) | preserved across refresh until explicit sign-out | Implicit in Supabase JWT claims |

A user is re-MFA-challenged only when they sign out explicitly, when the refresh token expires (rare), or when their session is cleared (incognito mode, manual localStorage wipe, etc.).

## Passkey / WebAuthn â€” deferred

Supabase Auth currently supports only `totp` and `phone` as MFA factor types. Native WebAuthn (passkey) support is being worked on but not yet GA. We intentionally don't build custom WebAuthn or a trusted-device cookie scheme; we'll revisit when Supabase ships native support.

## Appendix A â€” Why the abandoned plan didn't work

The earlier `v1.3-flexible-mfa-design.md` assumed Supabase's `mfa.enroll` accepted `factorType: 'webauthn'`. A quick `search_docs` lookup of the current docs confirmed it doesn't â€” the documented factor types are `totp` and `phone` only. The plan would have failed at the first API call.

The honest version of this work waits for native Supabase support and ships the smaller, real items now.

## Appendix B â€” What the user originally asked

> *"i want to add a remember me check in the login part of the app so whenever i want to come back i don't have to log in from the start."*

After clarification, this turned out to mean "skip the MFA code on return visits for a while." Supabase's default session behavior already satisfies this: AAL2 is preserved across token refreshes; the user only re-MFAs after an explicit sign-out or refresh-token expiry. Documenting this in `DEPLOYMENT.md` (Section 7 above) closes the loop without building UI that wouldn't change behavior.
